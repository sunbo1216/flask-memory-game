# 基于Flask阿尔茨海默记忆训练小游戏——“我可以倒着记” 实现要求
## 25-26-2学期《Web程序设计》期中考

## 一、项目简介
完成**前后端分离**的老人记忆康复训练Web小游戏开发，后端基于Flask技术栈实现接口服务与数据持久化，前端实现适老化交互页面、用户权限体系与完整游戏业务逻辑，实现注册登录、云端数据同步、离线本地缓存、记忆闯关训练、答题记录统计等核心功能。

“我可以倒着记”的游戏需求如下：【请根据课堂了解的需求填写】
1. 老年用户注册登录后开始游戏；
2. 游戏采用“数字倒序记忆训练”规则：系统先展示一串数字，用户需要在数字消失后，将看到的数字倒着输入。例如系统显示 358，用户应输入 853
3. 游戏设置连续闯关机制，用户不能手动选择关卡，必须从第 1 关开始逐步完成。当前关卡答题正确后自动进入下一关，答题错误则保留当前关卡或重新开始本关训练。
4. 游戏难度随关卡提升逐步增加：
第 1 关：记忆 2 位数字；
第 2 关：记忆 3 位数字；
第 3 关：记忆 4 位数字；
第 4 关：记忆 5 位数字；
第 5 关及以后：记忆 6 位数字；
最高难度不超过 6 位，避免对老年用户造成过高认知负担。
5. 每一关包含“数字展示阶段”和“用户答题阶段”。数字展示期间，用户只能观察数字，不能提前输入；数字隐藏后，输入框和提交按钮才可用
6. 游戏界面必须包含暂停按钮。点击暂停后，游戏计时停止，数字展示和答题流程暂停，暂停按钮文字变为“继续”。再次点击“继续”后，游戏恢复
7. 暂停状态下允许用户直接退出游戏。用户退出时，系统需要自动保存本次训练记录，并生成阶段性训练报告，不能因为中途退出导致记录丢失
8. 游戏过程中需要实时记录用户每一次作答信息，包括当前关卡、原始数字、正确倒序答案、用户输入内容、是否答对、作答用时和作答时间
9. 游戏数据采用本地与云端双存储机制。每次作答完成后，前端先将记录保存到 localStorage，再调用后端接口同步到 SQLite 数据库，保证断网或刷新页面时数据不丢失
10. 用户重新进入游戏时，系统优先读取本地缓存中的游戏进度；如果本地没有缓存，则自动向后端请求云端存档，恢复用户上次训练状态
11. 用户点击“退出游戏”或完成最高难度训练后，系统自动进入报告页面，展示本次训练结果统计，包括总作答次数、正确次数、错误次数、准确率、最高通关关卡、平均用时和完整答题明细
12. 前端页面采用适老化设计，要求字体较大、按钮明显、交互简单、颜色对比清晰，减少复杂操作，保证老年用户能够独立完成训练
13. 每次用户点击“开始训练”时，系统自动创建一个本次训练会话编号 session_id，用于区分不同训练批次。本次训练中的所有答题记录、关卡进度和训练报告都与该 session_id 绑定，保证报告统计的是本次训练数据，而不是混合历史数据
14. 考虑到老人记忆力有限只有六位数字的反复训练可能难度过大，因此增加训练模式，可以自己选择训练的数字长度

## 二、限定技术栈
### 1. 后端技术
- 核心框架：`Flask`
- 数据库：`SQLite` 
- ORM框架：`Flask-SQLAlchemy`
- 身份认证：`Flask-JWT-Extended`
- 密码加密：`Flask-Bcrypt`
- 跨域处理：`Flask-CORS`

### 2. 前端技术
- 基础技术：原生 HTML + CSS + JavaScript
- 存储方案：`localStorage` 离线本地存储
- 通信方式：原生 Fetch 请求对接 RESTful 接口

### 3. 其他要求
注意代码规范：注释清晰、变量命名规范、无冗余无效代码

6. **权限与安全**
   - 用户密码通过 `bcrypt` 哈希加密存储，禁止明文保存
   - 基于 JWT 实现无状态登录认证，登录接口下发 Token
   - 所有游戏业务接口添加 `jwt_required` 鉴权，未登录禁止访问


6. **接口功能实现**【继续填写你需要的接口定义】
   | 接口路径                       | 请求方式 | 功能描述                          |
   | -------------------------- | ---- | ----------------------------- |
   | `/api/users`               | POST | 用户注册，完成用户名、密码非空校验和重复用户名校验     |
   | `/api/auth/login`          | POST | 用户登录，校验账号密码，登录成功后下发 JWT Token |
   | `/api/auth/me`             | GET  | 获取当前登录用户信息，用于前端校验登录状态         |
   | `/api/game/progress`       | GET  | 获取当前用户的游戏进度                   |
   | `/api/game/progress`       | PUT  | 更新当前用户的关卡状态、最高关卡和统计数据         |
   | `/api/game/records`        | POST | 保存用户单次作答记录                    |
   | `/api/game/records`        | GET  | 获取当前用户的全部作答记录或本次训练记录          |
   | `/api/game/reports`        | POST | 退出游戏或完成训练后生成训练报告              |
   | `/api/game/reports/latest` | GET  | 获取当前用户最近一次训练报告                |

7. **异常处理**
   - 至少增加请求参数非空校验、字段完整性校验
   - 全局捕获JWT无效、Token过期、参数格式错误等异常
   - 自定义异常提示文案，前端可精准捕获错误信息

## 四、前端开发要求
1. **页面布局规范**
   - 包含：登录、注册、游戏启动、游戏运行、报告
   - 登录、注册页面**纯纵向垂直布局**，元素居中排列
   - 适老化设计：大字体、大按钮、宽间距、高辨识度配色、圆角控件

2. **用户权限功能**
   - 实现用户账号注册、密码登录、退出登录功能
   - 登录成功后JWT Token持久化存储至本地缓存
   - 页面初始化自动校验登录状态，未登录强制跳转登录页

3. **缓存与数据同步**
   - 采用「本地缓存优先」策略：优先读取离线游戏数据
   - 无本地缓存时，自动请求服务端接口拉取云端存档
   - 游戏进度、答题记录实时双向同步：本地`localStorage` + 服务端数据库

4. **报告与统计要求**
   - 结束游戏后自动生成训练报告
   - 统计指标：总作答次数、正确次数、答题准确率、最高通关关卡
   - 展示全部作答明细，包含作答类型、用时、对错标识
   - 报告数据保证用户此次游戏数据完整性
